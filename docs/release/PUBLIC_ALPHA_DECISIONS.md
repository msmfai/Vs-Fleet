# Public Alpha Decisions

Fleet is not ready to publish until every **Required before public alpha** item
has an explicit owner decision. These are the choices that are expensive to
change once strangers clone, package, fork, or depend on the project.

Answer the owner-facing worksheet in
[PUBLIC_ALPHA_OWNER_PROMPT.md](PUBLIC_ALPHA_OWNER_PROMPT.md), then record the
actual selected choices in [OWNER_DECISION_RECORD.md](OWNER_DECISION_RECORD.md).
Use [OWNER_RELEASE_APPROVAL.md](OWNER_RELEASE_APPROVAL.md) as the short-form
approval sheet and roughness check before marking the owner record approved.
Run `./scripts/public-alpha-decision-packet.sh` to list the current unresolved
owner choices and the follow-up commands implied by the selected history and
distribution answers.
Run
`./scripts/draft-owner-decisions.sh <github-owner> <github-repo> docs/release/OWNER_DECISION_RECORD.draft.md`
if you want a PENDING draft with the recommended source-alpha defaults checked
before making final choices.

## Required before public alpha

| Decision | Current state | Why it matters | Recommended alpha default |
|---|---|---|---|
| License | `MIT OR Apache-2.0` is applied in Rust crates, VS Code package manifests, package lockfiles, and root license files. `docs/release/LICENSE_INTENT.md` records the source-alpha posture and the forward-relicensing caveats. | Outsiders need clear rights to use, fork, package, and contribute. Changing later can be messy if external contributors appear. | Keep MIT/Apache-2.0 dual licensing for permissive Rust ecosystem norms. Keep reusable library/API crates permissive; reserve AGPL-3.0-only plus a commercial exception as a future CLI/hosted-control-plane contingency only after a concrete monetization trigger. Run `scripts/check-license-decision.sh` after the owner record is approved to verify every manifest and lockfile agrees. |
| Release scope | Mixed source tree: protocol/hub/reporter/CLI, macOS Tauri host, VS Code extensions, container eval harness, large design spec. | Users need to know what is usable today vs research/eval scaffolding. | Alpha scope should be "local macOS Fleet host + local code serve-web sessions + reporter/bridge pieces"; mark remote/deploy/container eval as experimental or internal; after choosing, run `scripts/check-alpha-scope-decision.sh` so public docs match the owner record. |
| Editor server boundary | The local source alpha launches the user's `code serve-web`; remote/deployed design docs discuss code-server/openvscode-server, Open VSX, and Microsoft VS Code Server licensing constraints. | Public users need to know Fleet is not redistributing Microsoft server or Marketplace components, and remote/deployed support should not accidentally depend on a proprietary server license. | For source alpha, use user-provided VS Code only and explicitly state no Microsoft VS Code Server or Marketplace redistribution; before remote/container support, require an OSS server/Open VSX path. After choosing, run `scripts/check-editor-server-boundary-decision.sh`. |
| Distribution | Source quickstart and release process are present; release bundle can be built locally; no signed/notarized macOS release policy. | Unsigned macOS apps trigger trust warnings. Public binary releases imply update/signing/support expectations. | Source-only alpha first. Add signed binaries later; after choosing, run `scripts/check-distribution-decision.sh` so package fences and binary-process docs match the owner record. |
| Artifact and secret policy | Raw host/eval artifacts are ignored going forward, but `scripts/history-release-check.sh` still finds prior local artifacts in branch history until history is cleaned or explicitly accepted. `scripts/check-public-release-branch.sh` verifies the recommended cleaned-history branch, public branch evidence, secret scan, and aggregate release gate against the public ref. | Public repos should not leak local paths/process details, credentials, or failed release evidence. | Before first GitHub publish, either squash/rewrite the branch so removed artifacts never appear in public history, or accept that old commits expose local paths. If cleaned history is selected, fill `docs/release/PUBLIC_BRANCH_EVIDENCE.md` for the exact source/public branch pair and run `scripts/check-public-release-branch.sh <public-branch> <source-ref-sha>`. Credential-looking findings are not an owner-accepted exception path; clean or rewrite them before public visibility. Keep only curated, redacted screenshots in public history. |
| Security disclosure | `SECURITY.md` is present; GitHub Private Vulnerability Reporting still needs to be enabled or replaced with a private contact before publish. | Users need to know how to report issues and what versions are supported. | Enable GitHub Private Vulnerability Reporting, or document a private contact channel before public visibility; after choosing, run `scripts/check-security-reporting-decision.sh` so the owner record and `SECURITY.md` agree. |
| Privacy/logging posture | Host/reporter logs contain workspace paths, command lines, session labels, and local URLs. | This product observes developer environments; privacy expectations must be explicit. | State that Fleet is local-first, has no telemetry by default, and logs local metadata that users should scrub before sharing; after choosing, run `scripts/check-privacy-decision.sh` so public docs match the owner record. |
| Contribution policy | `CONTRIBUTING.md`, `DCO.md`, and a PR template are present; broad outside code PRs are deferred until license choice is final. | First outside PR forces a licensing/provenance decision. A DCO preserves provenance but not unilateral relicensing rights over contributor-owned code. | Require DCO sign-off for outside code contributions; no CLA for source alpha. Revisit CLA before accepting code if commercial exceptions or proprietary relicensing become a goal. After choosing, run `scripts/check-contribution-decision.sh` so the docs and PR template agree. |
| Public name/namespace | Product and package names are `Fleet`, `fleet-*`, publisher `fleet-team`, bundle id `dev.fleet.host`. | Names collide easily and package namespaces can be hard to migrate. | Confirm GitHub org/repo, VS Code/Open-VSX publisher, crates package names, and macOS bundle identifier before publishing packages; after recording the choice, run `scripts/apply-namespace-decision.sh` for manifest metadata and `scripts/check-namespace-decision.sh` to verify it. |
| Support boundary | `SUPPORT.md` states best-effort alpha, breaking changes expected, and no production support/SLA. | Alpha users need expectations; otherwise every bug can become implied support. | Keep support best-effort for source-only alpha; after choosing, run `scripts/check-support-decision.sh` so support docs match the owner record. Revisit before binaries or package publishing. |
| Branding stability | Generated icon and possibly temporary `Fleet` name. | Public assets become recognizable quickly, and users may screenshot, fork, or write docs against them. Asset provenance also has to be concrete before publication. | Decide whether the `Fleet` name and icon are alpha placeholders or stable public assets before the first GitHub pre-release; either affirm `crates/fleet-host/icons/icon.png` may be redistributed under the chosen project license or replace it, then run `scripts/check-branding-decision.sh`. |
| Versioning and compatibility | Release notes say no stable upgrade path, stable APIs, or backwards-compatible state formats are promised during alpha. | Users can build automation around CLI flags, protocol schemas, and state files faster than expected. Backing away later is painful. | Use `v0.1.0-alpha.N` pre-release tags and promise no stable API/protocol/state/upgrade compatibility until the project has real outside users; after choosing, run `scripts/check-versioning-decision.sh`. |
| Community intake and moderation | Issue templates narrow public intake to scoped bug reports and alpha feedback; blank issues are disabled; the code of conduct gives maintainers moderation powers. | Opening public project spaces too broadly creates support, moderation, and security-reporting obligations before the alpha has capacity for them. | Open only scoped bug and alpha-feedback templates for source alpha, keep blank issues disabled, and keep Discussions off unless explicitly enabled later; after choosing, run `scripts/check-community-intake-decision.sh`. |
| Release custody and maintainer authority | GitHub publication evidence records branch protection and settings, but release tag and artifact authority still needs an explicit owner decision. | Public source tags, GitHub releases, package credentials, and repo settings are supply-chain controls. Ambiguity here becomes a security and governance problem fast. | Single-maintainer alpha: only the repository owner or named maintainer can push release tags, create GitHub releases, change repo settings, or publish packages; no package publishing credentials for source-only alpha. After choosing, run `scripts/check-release-custody-decision.sh`. |
| AI-assisted contribution provenance | The project itself is built around AI coding workflows, but outside AI-assisted patches need explicit provenance and privacy rules. | Contributors can accidentally submit model output they cannot license, private prompts/transcripts, local logs, workspace paths, or unreviewed generated code. | Allow AI-assisted contributions only when the contributor certifies human review, right to submit, and no private prompts, transcripts, logs, paths, or generated artifacts; after choosing, run `scripts/check-ai-contribution-decision.sh`. |
| Supported platform and toolchain | Quickstart currently documents macOS, Rust 1.78+, Node.js 20/npm, Git, and user-provided VS Code `code` CLI. | Public users infer support from install instructions. If Linux/Windows/remote paths look available but are not tested, support debt starts immediately. | State macOS source alpha only with the named toolchain; do not imply Linux, Windows, remote/container, or binary-package support until there is a support matrix and verification evidence. After choosing, run `scripts/check-platform-support-decision.sh`. |
| Public roadmap and non-goals | The issue templates invite alpha feedback, and future milestones may be useful for triage, but no public roadmap policy exists yet. | Sharp users treat issues, labels, and milestones as signals about what maintainers intend to deliver. If those are only triage notes, say so before they become perceived commitments. | Make no roadmap commitments during alpha: issues, labels, and milestones are triage hints, not delivery promises. Publish a real roadmap only if the owner is ready to maintain it; after choosing, run `scripts/check-roadmap-decision.sh`. |
| Public name collision and trademark posture | `Fleet` is a crowded developer-tool name and JetBrains has used Fleet for a developer IDE. The namespace and branding gates do not by themselves prove legal clearance. | A public repo name, app label, package prefix, screenshots, and forks can make a working name look like a stable trademark claim. Renaming later gets harder after users write docs or install packages. | Treat `Fleet` as a provisional source-alpha working name, make no trademark claim, and publish no stable packages or binaries under Fleet namespaces until the owner either clears the name or renames it; after choosing, run `scripts/check-name-collision-decision.sh`. |
| Local data and uninstall policy | Fleet writes Hub runtime files, host logs, spawned workspaces, VS Code server userdata, reporter sockets, and shim files under local runtime directories. | Public users need to know what Fleet puts on their machine, what survives quit/restart, and how to remove source-alpha state without assuming external servers are owned by Fleet. | Document `~/.fleet/run`, `~/.fleet/mux`, `FLEET_RUNTIME_DIR`, `FLEET_MUX_DIR`, manual cleanup, and process ownership boundaries; after choosing, run `scripts/check-local-data-decision.sh`. |
| GitHub Actions supply-chain posture | CI and release-readiness workflows use third-party Actions by tag and run on public pull requests once the repo is public. | Workflow token permissions, secrets, package credentials, and third-party Action pinning are supply-chain controls. Ambiguity here becomes a release/security problem when outside PRs appear. | For source alpha, accept tagged third-party Actions only with read-only `GITHUB_TOKEN` permissions, no secrets, and no publishing credentials. Require full SHA pinning before binaries/packages or if the owner wants stricter public CI; after choosing, run `scripts/check-workflow-supply-chain-decision.sh`. |
| Code of conduct | A short project-specific `CODE_OF_CONDUCT.md` is present. | Public issues/PRs need moderation expectations. | Use the short policy for alpha; switch to a standard covenant later if the community grows. |
| Public CI evidence | Normal CI and manual Release Readiness workflows exist; `docs/release/PUBLIC_CI_EVIDENCE.md` is a pending evidence record. | "CI was green" is meaningless unless tied to an exact commit and run; after public release, users will judge the project by that evidence. | Require GitHub Actions green on the exact public commit for both normal CI and Release Readiness, generate the evidence with `scripts/generate-public-ci-evidence.sh`, then run `scripts/check-ci-evidence-decision.sh` so the evidence record matches the owner decision. |
| GitHub repository settings | `GITHUB_PUBLICATION_RUNBOOK.md` describes repository settings, but `docs/release/GITHUB_PUBLICATION_EVIDENCE.md` is still pending. | Repository visibility, issue intake, security reporting, branch protection, and release/package settings are easy to forget during the visibility flip. | Fill the evidence record for the exact GitHub repo and commit, then run `scripts/check-github-publication-evidence.sh` before changing visibility or cutting the first public pre-release. |
| Dependency review evidence | `docs/release/DEPENDENCY_REVIEW.md` defines manual cargo/npm/workflow review commands; `scripts/check-lockfile-policy.sh` requires the Rust, pnpm, and npm lockfiles needed for exact-commit review; `docs/release/DEPENDENCY_REVIEW_EVIDENCE.md` is a pending evidence record; `.github/dependabot.yml` covers version-update surfaces, but no automated license/advisory allowlist exists yet. | First public users inherit the dependency graph and any advisories/license surprises. | Run and record the dependency review for the exact public commit, or explicitly accept skipping it in the owner record; after choosing, run `scripts/check-dependency-review-decision.sh`. |

## Should decide before packaging binaries

| Decision | Current state | Why it matters | Recommended alpha default |
|---|---|---|---|
| macOS signing/notarization | Local app bundle only. | Unsigned binaries are difficult for non-developers to run. | Do not ship binaries until Apple Developer ID signing and notarization are automated. |
| Update channel | None. | Auto-update is a security boundary and support commitment. | No auto-update in alpha. Use GitHub releases only. |
| Dependency/license review cadence | Manual dependency review is required for source alpha; no automated cargo/npm license allowlist is enforced yet. | Automation prevents drift after the first public release. | Add an automated allowlist/advisory policy after alpha if the project keeps taking outside users. |

## Explicit "do not forget" calls

- Public alpha is an external promise. If the README says a feature works, it
  should either have a quickstart path or be labeled experimental.
- Do not publish a repo with raw local path artifacts unless you are comfortable
  with that data being indexed forever.
- Do not accept outside contributions before the license/contributor policy is
  settled, and do not treat DCO as a substitute for a CLA if future commercial
  exceptions or proprietary relicensing become a goal.
- Do not ship a macOS binary until you decide whether "unsigned alpha" is
  acceptable for your audience.
